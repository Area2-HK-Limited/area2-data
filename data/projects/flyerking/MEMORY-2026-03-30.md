| **D** | **Sub-agent 錯誤選 channel/target** | XHS sub-agent 事故：sub-agent 直接發圖，唔經 orchestrator 驗證，錯發帖子封面當 QR code |

### 即時行為規則（每次 send 圖片必須跟）

1. **用 `filePath` 而非 `mediaUrl`，路徑必須喺 workspace**
   - ✅ `/home/openclaw/.openclaw/workspace/data/...`
   - ✅ `/tmp/openclaw/...`（image-generation skill 用）
   - ❌ `/tmp/xxxxx.png`（delivery queue 唔 persist，restart 後 file 消失）

2. **回圖必須跟 inbound channel**
   - 收到 WhatsApp → `message(channel="whatsapp", target=<同一個 JID>)`
   - 收到 Telegram → `message(channel="telegram", target=<同一個 chat_id>)`
   - ⚠️ 唔好硬 code channel；用 inbound session context 嘅 channel

3. **tool success ≠ 用戶收到**
   - WhatsApp stale-socket 期間，`Sent message 3EB0...` 喺 log 係假成功
   - 如果 Eric 話「睇唔到」→ 重發，唔好爭論 tool 話 success

4. **Sub-agent 唔可以直接發媒體俾用戶**
   - Sub-agent 負責生成 → 通知 orchestrator → orchestrator 驗證後發送
   - 唯一例外：明確授權（例如 watchdog cron）

### 🐕 Watchdog Cron 正確做法 (2026-03-07) ⭐⭐

**錯誤做法 ❌：**
```bash
openclaw cron add --session main --system-event "..." 
```
→ `--session main` + `--system-event` 唔 work，watchdog 唔會觸發任何通知。

**正確做法 ✅：**
```bash
openclaw cron add \
  --name "watchdog-xxx" \
  --every 5m \
  --session isolated \
  --announce \
  --to <CHAT_ID> \
  --message "Watchdog: 讀 /tmp/task-xxx.json，status != done → 發警報..."
```

**關鍵：**
- `--session isolated`（唔係 main）
- `--announce` + `--to <chatId>`（必須同時填）
- ErIc Telegram chat ID：`5989270324`

### Channel-Aware Media Send (2026-03-21) ⭐
- 同一個人喺 WhatsApp / Telegram / Discord 係**不同 channel session**；回覆圖片/文件時必須跟返當前 inbound channel。
- **唔可以**因為係同一個人就跨 channel 補發，亦唔可以沿用舊 channel / 舊 target / 內部通知 channel。
- Gateway / tool 顯示 `messageId`、`tool success`、`Sent via ...` 只代表系統接受咗發送，**唔等於用戶已實際見到媒體**。
- 交付完成標準：用戶實際喺當前 channel 見到圖片/文件；如果用戶話收唔到，就視為未完成。
- 如需發媒體，優先將檔案放入 workspace `data/` 穩定路徑，再按當前 inbound metadata 的 channel/account/chat target 發送。

## 2026-02-13: B03 Batch Image Processor 事故修復

### 問題
B03 Agent 處理大量發票圖片（7z 壓縮檔）時完全停止回覆。用戶傳咗一個含大量圖片嘅檔案，Agent 處理到一半就死咗。

### 根本原因（多個）
1. **Batch size 太大** — 設定為 8，但 conversation context 會累積歷史圖片，實際發送了 11-13 張圖片，超過 Gemini 3 Pro 的 10 張限制
2. **大量 `400 too many images` 錯誤** — 多個 subagent 同時報錯
3. **Gateway timeout (60s)** — 匯總回覆時超時
4. **Embedded run timeout (600s)** — 整個 Agent 10 分鐘超時後死咗
5. **Rate limit** — `Profile github-copilot:github timed out (possible rate limit)`
6. **冇 retry/fallback 機制** — 失敗後就無法恢復

### 修復
1. **降低 batch size** — Gemini: 8→5, Claude Sonnet: 16→10 (留 50% buffer)
2. **獨立批次資料夾** — 每批圖片放入 `batches/batch_N/` 資料夾，避免 context 累積
3. **並行限制** — 最多 3 個 subagent 同時跑
4. **支持 7z/rar** — 自動偵測壓縮格式
5. **增加錯誤重試機制** — too many images 自動切半重試
6. **增加 timeout 保護** — 單個 batch 120秒，整體 8 分鐘
7. **進度回報** — 大量圖片時定期通知用戶

### 教訓
- Model 的圖片限制要留 50% buffer，唔係 20%
- Conversation context 會累積歷史圖片，佔用限額
- 寧願分多幾批慢啲，都唔好一次塞太多導致全部失敗
- 並行 subagent 太多會觸發 rate limit 同 gateway timeout

## 🖼️ Batch Image Processor 經驗 (2026-02-14)

### Batch Size 優化
- **Gemini batch size 5→3** — context 累積導致 "too many images: got 15"（5 張 × 3 倍累積）
- 留 70% buffer 比舊嘅 50% 更保守

### 分組 Spawn 規則
- ⛔ NEVER spawn all subagents at once
- Spawn 3 → wait → spawn 3 → wait
- 一 pass 處理慳 50% token（subagent 直接回覆 JSON array，主 agent parse + 匯總）
- **測試結果：** 85 張圖片 → 29 批（3×28 + 1×1），驗證通過

### Batch 後處理清理
- 舊 `/tmp/receipts_process/` 數據污染新結果 → 自動清理超過 30 分鐘嘅舊 temp 目錄
- 永遠唔好重用舊路徑或讀舊結果檔
- 用唯一時間戳路徑 `/tmp/receipts_$(date +%s)`

### ⚠️ 主 Agent 唔可以讀圖片 ⭐⭐⭐
- 主 Agent (Gemini) 嘅 conversation context 會累積所有歷史圖片
- Subagent 有獨立 context，讀完就釋放
- 所以圖片處理**必須**由 subagent 做，主 Agent 只做 orchestrator
- 主 Agent 自己讀圖片 → context 爆炸（got 12 → 15 → 18 → 21 → 24 → 27 → 39）

### 🗜️ 圖片壓縮優化 — PNG→JPG + Resize 1024px
- 所有 PNG/WebP/BMP/GIF → JPG (quality=85, -strip metadata)
- Resize 到寬度 ≤1024px（`-resize 1024x>`，只縮小唔放大）
- PDF 轉換改用 JPG：`pdftoppm -jpeg -jpegopt quality=85 -r 120`
- **效果：** 平均 750KB→116KB/張 (85%↓)，總大小 220MB→10MB (95.5%↓)，Token 60-70%↓
