## 1. 背景與目標

### 1.1 項目背景

PersonAI 由 **Hostlink (HK) Limited** 與 **Area2 (HK) Limited** 聯乘推出：

| 公司 | 角色 | 負責範疇 |
|------|------|---------|
| **Hostlink (HK) Limited** | 技術夥伴 | 開發、系統架構、資訊安全、伺服器基礎設施 |
| **Area2 (HK) Limited** | 產品及市場夥伴 | 產品設計、包裝、宣傳、市場推廣 |

**核心優勢：** Hostlink 提供企業級亞太區基礎設施（Tencent Cloud ISO 27001 / SOC 2 / CSA STAR 認證底層，部署於日本地區），Area2 提供精準香港 SME 數碼營銷能力——兩者互補，為香港中小企帶來「信得過、用得順」的 AI 助手解決方案。

### 1.2 核心問題（解決咩痛點）

**SME 嘅真實痛點（基於 LIHKG / XHS / HKGolden 用戶聲音）：**

| 痛點 | 現況 | PersonAI 解決方案 |
|------|------|------------------|
| 想用頂級 AI，但 ChatGPT / Claude 香港封鎖 | 需要 VPN，複雜又有法律風險 | 直接 WhatsApp 廣東話使用 GPT-5.4 + Claude + Gemini |
| AI 每次都唔記得我係誰 | 每次重頭解釋 | 永久記憶，越用越識你 |
| 訂了一堆工具，零散又貴 | 每月 HK$2,000+ 訂閱費 | 一個平台，60+ 技能全包 |
| 怕業務資料外洩，唔敢用 AI | 數據去咗美國，受美國法律管 | 用戶對話記錄儲存亞太區（日本），唔做廣告、唔訓練模型，PDPO 合規 |
| 唔識用，無人問 | 純機器 Chatbot，問完都唔知 | 真人 WhatsApp 廣東話技術支援 |

**目標用戶：** 香港中小企老闆及管理層（餐飲 / 零售 / 地產 / 會計 / 教育 / 美容 / 物流行業為優先）

## 4. 每頁 Wireframe 描述及設計規範

### 4.1 全站視覺設計規範

**色彩系統（Dark Theme — 參考 zeabur.com/zh-TW/ 風格）：**
| 顏色 | 用途 | HEX |
|------|------|-----|
| 主背景（Deep Dark） | 頁面底色 | `#0A0A0F` / `#0D1117` |
| 交替 Section 底色 1 | 次要 Section | `#111827` |
| 交替 Section 底色 2 | 特色 Section | `#0F172A` |
| 特色 Section 漸層 | 重點 Section 背景 | `#1E1B4B → #0F172A`（深藍漸層）|
| 文字（Body） | 正文 | `#E2E8F0`（淺灰白）|
| 文字（Muted） | 次要說明 | `#94A3B8` |
| 主色（Primary Blue） | CTA 按鈕、重點標題、icon | `#3B82F6`（亮藍）|
| Accent 漸層 | 特色標題、漸層按鈕 | 紫 `#8B5CF6` → 藍 `#3B82F6` → 青綠 `#06B6D4` |
| 卡片背景 | Feature / Pricing 卡片 | `#1E293B` |
| 卡片 Border | 卡片邊框 | `#334155` |
| 成功 / 確認 | 安全承諾 icon | `#22C55E` |
| 強調橙 | 重點數據（如 HK$298）| `#F97316` |

> ⚠️ **全站深色背景（Dark Theme）。** Section 交替使用 `#0A0A0F` / `#111827` / `#0F172A`，特色 Section 用深藍漸層。CTA 按鈕用漸層或亮色對比。Nuxt config：`colorMode.preference: 'dark'`。

**字體：**
- 標題：`Noto Sans TC` Bold（Google Fonts，支援繁體中文）
- 正文：`Noto Sans TC` Regular
- 英文代碼/數字：`Inter` Regular

**圖片使用規範：**
- 每個主要 Section 都要有相關圖片（右或左交替配圖）
- 圖片風格：**3D Pixar 風格插圖 / 現代扁平設計 / 真實辦公室場景** — 三種混合
- **所有圖片由 `skills/image-generation`（Nano Banana Pro via Google AI Studio API）生成** — 唔用 stock photo，唔用 DALL-E
- 圖片格式：WebP（優先），JPG fallback
- 所有圖片必須有 alt text（SEO + 無障礙）

**💅 美容業（Beauty）**

**📦 物流業（Logistics）**
- 痛點：派送更新要逐個 WhatsApp；貨物狀態查詢壓垮客服；司機調度記唔住

### 7.4 i18n 多語言設定規格（v1.6a 新增）

**Module：** `@nuxtjs/i18n`（Nuxt 官方 i18n 模組）

| 設定項 | 值 |
|--------|-----|
| **支援語言** | `zh-TW`（繁體中文，預設）、`en`（English）|
| **預設語言** | `zh-TW` |
| **URL 策略** | `prefix_except_default`（例：`/pricing` = 繁中；`/en/pricing` = 英文）|
| **語言切換 UI** | Header 右上角 toggle：`繁中 / EN` |
| **翻譯檔位置** | `locales/zh-TW.json`、`locales/en.json` |
| **HTML lang** | `<html lang="zh-TW">` / `<html lang="en">`（自動切換）|
| **SEO hreflang** | 自動注入 `<link rel="alternate" hreflang="zh-TW" ...>` + `<link rel="alternate" hreflang="en" ...>` |

**`nuxt.config.ts` 相關設定：**

```typescript
export default defineNuxtConfig({
  modules: [
    '@nuxt/ui',
    '@nuxt/content',
    '@nuxtjs/i18n',
  ],
  colorMode: {
    preference: 'dark',   // 強制深色主題
    fallback: 'dark',
  },
  i18n: {
    locales: [
      { code: 'zh-TW', name: '繁體中文', file: 'zh-TW.json' },
      { code: 'en', name: 'English', file: 'en.json' },
    ],
    defaultLocale: 'zh-TW',
    strategy: 'prefix_except_default',
    lazy: true,
    langDir: 'locales/',
  },
})
```

### 18.4 技術規格

#### 18.4.1 表單數據儲存方案

**方案：自建 PostgreSQL + PostgREST（基於 `Area2-HK-Limited/nuxt-dashboard-starter`）**

| 技術組件 | 版本 | 說明 |
|---------|------|------|
| **PostgreSQL** | 16 | 關係型資料庫，儲存所有登記記錄 |
| **PostgREST** | v12 | 自動將 PostgreSQL schema 轉為 RESTful API |
| **Adminer** | 最新版 | Admin dashboard，暫代 CMS 查看記錄 |
| **Docker Compose** | — | 一鍵部署整個 stack |
| **Nuxt Server API** | — | Proxy layer，連接前端同 PostgREST |

**Starter Repo：** `Area2-HK-Limited/nuxt-dashboard-starter`
- 已有 PostgreSQL + PostgREST + Adminer + Docker Compose 配置
- 已有 CRUD 功能，直接複用

**資料庫 Schema：`early_access_registrations` table**

```sql
CREATE TABLE early_access_registrations (
  id              SERIAL PRIMARY KEY,
  name            TEXT NOT NULL,
  company_email   TEXT NOT NULL UNIQUE,
  whatsapp        TEXT NOT NULL,
  company_name    TEXT NOT NULL,
  ai_use_cases    TEXT[] DEFAULT '{}',  -- 最想AI做咩（array）
  other_needs     TEXT,                  -- 其他需求（自填）
  utm_source      TEXT,
  utm_medium      TEXT,
  utm_campaign    TEXT,
  seq_number      INTEGER,               -- 登記序號（trigger 自動填入）
  status          TEXT DEFAULT 'registered' CHECK (status IN ('registered', 'waitlist')),
  agreed_terms_at TIMESTAMPTZ NOT NULL,  -- 同意條款時間
  created_at      TIMESTAMPTZ DEFAULT NOW()
);
```

**Nuxt Server API 端點：**

```
POST /api/early-access/register
```
→ 驗証欄位（公司 email 域名黑名單）
→ 查詢目前 count（SELECT COUNT(*) FROM early_access_registrations WHERE status = 'registered'）
→ count ≤ 50：status = 'registered'；count > 50：status = 'waitlist'
→ INSERT via PostgREST REST API → PostgreSQL
→ 建立 Stripe Checkout Session（`server/api/stripe/checkout.post.ts`），plan 按登記方案
→ 回傳 `{success: true, seq_number: N, status: 'registered' | 'waitlist', stripe_checkout_url: 'https://...'}`
→ 前端 redirect 到 stripe_checkout_url

```
GET /api/early-access/count
```
→ 查詢 registered count + waitlist count
→ 加入 5 分鐘 server-side cache
→ 回傳 `{registered: N, waitlist: M, total: N+M}`

**名額管理邏輯（API 層）：**
- `registered` count ≤ 50：正常登記，status = 'registered'
- `registered` count > 50：自動標記 status = 'waitlist'，序號繼續遞增
- 前端根據 count API 即時更新進度條 + 按鈕文字

**Admin Dashboard：Adminer（暫代）**
- 存取 URL：`http://{server}:8080`（Adminer）
- 可直接瀏覽 / 匯出 `early_access_registrations` 表格記錄
- Phase 2 再考慮建立獨立 admin panel

### 18.5 Stripe Checkout 整合技術規格（v1.6a 新增）

#### 18.5.1 Stripe Checkout 流程架構

```
用戶點擊「開始 14 日免費試用」/ 提交 Early Access 表單
    ↓
POST /api/stripe/checkout
    body: { plan: 'starter' | 'business', email?, name? }
    ↓
Nuxt Server API:
    1. 建立 Stripe Checkout Session
       - mode: 'subscription'
       - payment_method_collection: 'if_required'
       - subscription_data.trial_period_days: 14
       - success_url: /early-access/success?session_id={CHECKOUT_SESSION_ID}
       - cancel_url: /pricing
    2. 回傳 checkout session URL
    ↓
前端 redirect 到 Stripe Checkout
    ↓
用戶完成（唔需要信用卡）
    ↓
Redirect 回 /early-access/success
```

#### 18.5.2 Stripe Webhook

**Endpoint：** `POST /api/stripe/webhook`

| 監聽事件 | 處理動作 |
|---------|---------|
| `checkout.session.completed` | 更新 PostgreSQL `early_access_registrations` registration status → 'activated' |
| `customer.subscription.trial_will_end` | 排程提醒用戶（Email + WhatsApp）：試用期將於 3 日後結束 |
| `customer.subscription.deleted` | 更新用戶狀態為 'cancelled' |
| `invoice.payment_failed` | 通知用戶付款失敗，引導更新付款方式 |

## Chapter 19：AI 模型開發分工方案（Development Workflow with Multi-Model AI）

### 19.1 三模型角色定位

PersonAI 網站開發採用三個 AI 模型各司其職嘅分工策略，按模型專長分配唔同階段嘅工作：

| 模型 | 專長 | 角色 | 一句話 |
|------|------|------|--------|
| **GPT-5.4** | 系統思維、Agentic 規劃、用戶心理、創意文案 | 🏗️ 架構師 + 🎨 UX 顧問 | 「佢識諗大局，企用戶角度」 |
| **Claude Sonnet 4.6** | 代碼生成最準、框架理解深、指令跟從度高、精準修改 | ⌨️ 主力開發 + 🔧 修正 | 「佢寫 code 最準，改得最精準」 |
| **Gemini 3 Pro** | 超長 context (1M tokens)、全局分析、系統性找 bug | 🔍 QA 總監 | 「佢一眼睇晒成個 project」 |

### 19.2 六階段開發流程

#### Phase 1：架構規劃 → GPT-5.4
- 定義 Nuxt 目錄結構、組件拆分策略

- 數據流規劃（PostgreSQL ↔ PostgREST ↔ Nuxt Server API ↔ Frontend）
- 技術選型決策（SSR vs SSG vs CSR 每頁分配）

- Nuxt Server API endpoints（register → PostgREST → PostgreSQL、count + waitlist logic、5分鐘 cache）
- Tailwind CSS / Nuxt UI 樣式實現
- TypeScript 類型定義
- 公司 Email 域名黑名單驗証邏輯

  - 表單欄位 placeholder 文字

  - Loading 狀態文字
  - 空狀態 / 名額已滿狀態

### 19.3 反模式（Anti-patterns）— 唔建議嘅做法

| ❌ 反模式 | 原因 | ✅ 正確做法 |
|---------|------|----------|
| 用 Gemini 寫 code | 容易 over-generate、格式唔一致、夾雜不必要嘅重構 | Sonnet 寫 code，Gemini 只做 review |
| 用 Sonnet 做 UX 決策 | 太 literal，缺乏創意跳躍，唔識「感覺」 | GPT 做 UX 決策，Sonnet 負責實現 |
| 用 GPT 做 QA | 容易漏 bug，唔夠 systematic，傾向「looks good」 | Gemini 做 QA（超長 context + systematic） |
| 一個模型做晒全部 | 浪費各模型專長，結果平庸 | 按階段分工，各展所長 |

### 19.4 開發效率 Tips

1. **Sonnet 開發時** → 俾佢完整 dev plan + wireframe 做 context → 輸出一致性大幅提升
2. **Gemini QA 時** → 俾佢整個 `src/` 目錄 + config files → 全局 bug 捕獲率最高
3. **GPT UX review 時** → 俾佢實際截圖 + wireframe 對比 → 建議最 actionable
4. **每個階段交接** → 產出一份結構化報告（Markdown），下一階段模型直接讀取
5. **版本控制** → 每個 Phase 完成後 git commit + tag，方便回滾
